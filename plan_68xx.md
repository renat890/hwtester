## Этап 1. Конфигурация

### 1.1 Определение структуры конфига

Описать Go-структуру, отражающую все параметры, задаваемые перед тестированием:

- Ожидаемый объём ОЗУ (в ГБ)
- Количество дисков и ожидаемый объём каждого
- Минимальная скорость чтения диска (МБ/с)

- Имена сетевых интерфейсов для Ethernet-теста

- Пороги температуры (макс. 85°C, градиент +40°C)
- Флаги включения/отключения опциональных тестов

### 1.2 Файл конфигурации по умолчанию

Создать `configs/default.yaml` со значениями по умолчанию из ТЗ.

### 1.3 Загрузка и валидация

1. Написать функцию `Load(path string) (*Config, error)`:
   - чтение файла
   - десериализация YAML
   - валидация (объём ОЗУ > 0, количество дисков > 0 и т.д.)
2. Предусмотреть мерж: значения из файла + переопределение через TUI.

### 1.4 Тесты (TDD)

- Тест загрузки корректного файла.
- Тест невалидных значений (отрицательный объём, пустой путь).
- Тест значений по умолчанию при частично заполненном файле.

## Этап 2. Абстракция тестов и раннер

### 2.1 Интерфейс теста

Определить единый контракт для всех проверок:

```go
type HWTest interface {
    Name() string
    Run(ctx context.Context) TestResult
}

type TestResult struct {
    Name     string
    Status   Status        // Pass / Fail / Skip / Error
    Duration time.Duration
    Details  string
    Metrics  map[string]any // произвольные метрики для отчёта
}
```

### 2.2 Раннер последовательного выполнения

1. Принимает `[]HWTest` и конфиг.
2. Выполняет тесты строго по порядку.
3. Поддерживает `context.Context` для отмены (Ctrl+C).
4. Собирает `[]TestResult` и передаёт в генератор отчёта.
5. Во время выполнения отправляет обновления в TUI через канал.

### 2.3 Тесты (TDD)

- Мок-тесты (Pass/Fail/Skip) — проверить порядок, сбор результатов, поведение при отмене контекста.

## Этап 3. Проверка ОЗУ

### 3.1 Получение объёма памяти

1. Использовать `gopsutil/mem.VirtualMemory()` для получения `Total`.
2. Обернуть вызов в интерфейс `hw.MemoryProvider`, чтобы можно было подменить в тестах.

### 3.2 Логика сравнения

1. Получить фактический объём.
2. Вычислить допуск: `expected * 0.98` … `expected * 1.02`.
3. Если фактический объём в допуске — Pass, иначе — Fail.
4. В `Metrics` записать: `expected_gb`, `actual_gb`, `deviation_pct`.

### 3.3 Тесты (TDD)

- Мок MemoryProvider: точное совпадение → Pass.
- Мок: отклонение +1.5% → Pass.
- Мок: отклонение +3% → Fail.
- Мок: отклонение −2% → Pass (граница).

---

## Этап 4. Проверка ПЗУ (обнаружение дисков)

### 4.1 Обнаружение дисков

1. Использовать `gopsutil/disk.Partitions()` и/или `disk.IOCounters()` для перечисления блочных устройств.
2. Дополнительно — чтение `/sys/block/` для получения списка SATA-дисков, фильтрация по типу (исключить loop, ram, nvme если не нужны).
3. Обернуть в интерфейс `hw.DiskDetector`.

### 4.2 Проверка количества и объёма

1. Проверить, что найдено ровно 4 диска (или число из конфига).
2. Для каждого диска получить объём (`disk.Usage()` или чтение `/sys/block/<dev>/size`).
3. Сравнить объём каждого с ожидаемым (допуск — на усмотрение, в ТЗ точное соответствие).

### 4.3 Тесты (TDD)

- 4 диска, все объёмы совпадают → Pass.
- 3 диска → Fail.
- 4 диска, один отличается по объёму → Fail.

## Этап 5. Проверка скорости диска

### 5.1 Подготовка

1. Проверить наличие файла `largefile` на каждом диске (или подготовить его, если конфиг это предусматривает).
2. Сброс кэша ОС: запись `"3"` в `/proc/sys/vm/drop_caches` (обязателен root).

### 5.2 Реализация замера

Рекомендуемый подход — нативная реализация через `os.File`:

1. Открыть файл/устройство на чтение.
2. Создать буфер 32 КБ.
3. Засечь время (`time.Now()`).
4. Читать в цикле до EOF, суммируя прочитанные байты.
5. Вычислить скорость: `totalBytes / elapsed`.

Альтернатива — вызов `dd` через `exec.Command`, парсинг вывода.

### 5.3 Критерий

- Скорость ≥ 500 МБ/с на каждом диске → Pass.
- Любой диск ниже порога → Fail (указать какой именно).

### 5.4 Тесты (TDD)

- Юнит-тест логики расчёта скорости по известным bytes/duration.
- Мок файла для проверки корректности чтения буфером 32 КБ.

## Этап 6. Проверка Ethernet (опциональный)

### 6.1 Предусловия

1. Проверить, что оба интерфейса из конфига существуют и имеют IP в одной подсети.
2. Если интерфейсы не найдены или тест отключён в конфиге — Skip.

### 6.2 Реализация iperf-подобного теста

**Вариант A — через iperf3:**

1. Запустить `iperf3 -s` на одном интерфейсе (горутина).
2. Запустить `iperf3 -c <ip> --json` с другого.
3. Распарсить JSON-вывод: потери, скорость.
4. Поменять направление и повторить.

**Вариант B — нативная реализация:**

1. Горутина-сервер: слушает TCP/UDP на интерфейсе A, считает принятые пакеты/байты.
2. Горутина-клиент: отправляет N пакетов фиксированного размера с интерфейса B.
3. Сравнить отправленное и полученное.
4. Поменять роли и повторить.

### 6.3 Критерий

- 100% доставка в обоих направлениях (порог уточняется).
- В `Metrics`: скорость, потери, направление.

### 6.4 Тесты (TDD)

- Мок сетевого соединения (net.Pipe) — передача без потерь → Pass.
- Имитация потери пакета → Fail.

---

## Этап 7. Проверка COM-портов (опциональный)

### 7.1 Предусловия

1. Проверить наличие двух COM-портов из конфига (`/dev/ttyS0`, `/dev/ttyS1` и т.д.).
2. Если порты не найдены или тест отключён — Skip.

### 7.2 Реализация

1. Открыть оба порта с параметрами: 115200 бод, 8N1.
2. Горутина A: пишет серию тестовых сообщений в порт 1.
3. Горутина B: читает из порта 2, сверяет с отправленным.
4. Поменять направление и повторить.
5. Библиотека: `go.bug.st/serial` или `github.com/tarm/serial`.

### 7.3 Критерий

- 100% сообщений доставлено в обоих направлениях.

### 7.4 Тесты (TDD)

- Мок serial-интерфейса (io.ReadWriter) — корректная передача → Pass.
- Обрыв соединения / таймаут → Fail.

## Этап 8. Проверка USB (опциональный)

### 8.1 Предусловия

1. Проверить, что USB-накопитель смонтирован по пути из конфига.
2. Если не найден или тест отключён — Skip.

### 8.2 Реализация

1. Обнаружить USB-устройства через `gopsutil/disk` или `lsblk`.
2. Проверить наличие файла `test.txt` на накопителе.
3. Прочитать файл, убедиться в отсутствии ошибок ввода-вывода.
4. Опционально: сверить контрольную сумму (если эталон задан в конфиге).

### 8.3 Критерий

- Устройство найдено + файл прочитан без ошибок → Pass.

### 8.4 Тесты (TDD)

- Мок файловой системы: файл существует и читается → Pass.
- Файл отсутствует → Fail.
- Ошибка чтения → Fail.

## Этап 9. Стресс-тест / проверка охлаждения

### 9.1 Мониторинг температуры

1. Использовать `gopsutil/host.SensorsTemperatures()` для получения текущей температуры.
2. Обернуть в интерфейс `hw.TempMonitor`.
3. Зафиксировать начальную температуру перед стартом нагрузки.

### 9.2 Нагрузка

**Вариант A — stress-ng:**

1. Запустить `stress-ng` через `exec.Command` с параметрами из ТЗ.
2. Читать вывод для логирования.

**Вариант B — нативные горутины:**

1. CPU-нагрузка: `runtime.NumCPU()` горутин с бесконечными вычислениями (напр. SHA-хэширование).
2. Память: периодическое выделение и заполнение буферов.
3. Диск: параллельная запись/чтение временных файлов.

### 9.3 Цикл мониторинга

1. Каждые N секунд (напр. 5) снимать показания температуры.
2. Передавать текущие показания в TUI для отображения прогресса.
3. Если температура > 85°C — досрочно остановить тест, Fail.
4. По истечении 10 минут: вычислить градиент (текущая − начальная).

### 9.4 Критерий

- Температура ни разу не превысила 85°C **И** градиент ≤ 40°C → Pass.
- Иначе → Fail.

### 9.5 Тесты (TDD)

- Мок TempMonitor: стабильная температура 60°C → Pass.
- Мок: температура растёт до 90°C → Fail (превышение порога).
- Мок: начало 30°C, конец 75°C (градиент 45°C) → Fail.

## Этап 10. Генерация отчётов

### 10.1 JSON-отчёт

1. Определить структуру отчёта:
   - Метаданные (дата, имя устройства, версия ПО).
   - Массив результатов тестов (имя, статус, метрики, длительность).
   - Итоговый статус (Pass только если все обязательные тесты пройдены).
2. Сериализовать через `encoding/json` с `MarshalIndent`.
3. Сохранить в файл с именем, включающим дату и серийный номер.

### 10.2 HTML-отчёт

1. Создать шаблон `templates/report.html` (Go `html/template`).
2. Цветовая индикация: зелёный — Pass, красный — Fail, серый — Skip.
3. Таблица с результатами каждого теста, развёрнутые метрики.
4. Сводка вверху: устройство, дата, общий результат.

### 10.3 Тесты (TDD)

- Генерация JSON из фиксированного набора результатов — проверить структуру и значения.
- Генерация HTML — проверить отсутствие ошибок шаблона, наличие ключевых строк.

## Этап 11. TUI (BubbleTea)

### 11.1 Экраны

1. **Стартовый экран:** выбор конфигурации, переопределение параметров, выбор опциональных тестов.
2. **Экран выполнения:** список тестов, текущий тест подсвечен, статус завершённых (✓/✗/—), прогресс-бар для стресс-теста, температура в реальном времени.
3. **Экран результатов:** итоговая таблица, путь к сохранённому отчёту.

### 11.2 Модель BubbleTea

1. `Model` хранит: конфиг, текущее состояние (экран), список тестов, результаты, канал обновлений от раннера.
2. `Update` — обработка клавиш (навигация, запуск, выход) и сообщений от раннера (тест начался, тест завершён, обновление температуры).
3. `View` — рендер текущего экрана через lipgloss.

### 11.3 Интеграция с раннером

1. Раннер работает в отдельной горутине.
2. Обновления передаются через `tea.Cmd` / каналы.
3. По завершении всех тестов — автоматический переход на экран результатов.

### 11.4 Тесты

- Тест Model.Update: отправка сообщений, проверка переходов между экранами.
- Тест View: корректный рендер для разных состояний.
